<objective>
Make failures safe and legible at the UI/orchestration layer. Today, when a
translation fails mid-flight, compose mode leaves whatever was typed and YOLO
can leave a half-state; the badge always shows the same red "ERR" regardless of
cause; and the background does no structured logging. This plan: (1) snapshots
the field before any replacement and restores it on failure in BOTH compose and
YOLO (ERR-03 user-visible recovery, no silent broken state), (2) surfaces the
error kind distinctly on the badge (ERR-04/05), and (3) adds structured
debug/error logging in the background service worker (LOG-01/02).

Purpose: Close the last Phase 2 gaps at the layer the user actually sees, using
the typed error kind that Plan 01 defined and Plan 02 attaches to thrown errors.

Output: hardened src/background.ts (logging + kind in response) and
src/content.ts (snapshot/restore + kind-aware badge), plus a human-verify
checkpoint for the live failure-recovery and distinct-error UX.
</objective>

| Boundary | Description |
|----------|-------------|
| background <-> content (chrome.runtime messages) | Error message + kind cross from background to content script |
| content script -> host page field | Snapshot text re-inserted into the user's field on failure via execCommand |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-07 | Information disclosure | background failure logging | mitigate | console.error logs provider/model/status/kind/endpoint/message only; API key is never read into the log object (key stays in translateText/provider scope) |
| T-02-08 | Tampering | snapshot restore writes into focused field | accept | Restore re-inserts the user's own original text via execCommand insertText (plain text, undo-safe); no external data inserted on the failure path |
| T-02-09 | Repudiation | logs only in local service-worker console | accept | LOG-01/02 are local debug aids per BYOK/no-backend architecture; no remote telemetry by design |
</threat_model>
