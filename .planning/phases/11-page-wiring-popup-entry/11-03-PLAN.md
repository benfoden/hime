<objective>
Wire the search page end-to-end: replace the `?state=` mock driver in search.ts with the live worker round-trip, add a Google-style top search bar + read-only disclosure line, and orchestrate the 3-stage progressive render (skeleton → raw Brave results → translated overlay) on the existing `renderSerp` seam. This is the page half of SRCH-02/SRCH-03 and XLT-05.

Purpose: search.ts currently only renders mock SERP states from `?state=`. This plan makes the page real: user types in the search bar → page calls `searchTranslated` (Plan 01 now returns `translatedQuery`/`translationFailed`) → disclosure line appears with skeleton (D-11) → raw Brave results render → `translateBatch` overlay replaces them. The disclosure line is read-only (D-07) with three phrasings (D-05 translated / D-06 direct / D-10 degraded). Re-submitting re-runs the pipeline in place (D-09).

Output: A pure `buildDisclosureText` helper (testable in node), search-bar + disclosure markup/CSS within the existing light theme, and a live-orchestration search.ts. Depends on Plan 01's extended `SearchTranslatedResponse`.
</objective>

  <action>
    Rewrite src/search.ts to drive the live pipeline behind the renderSerp seam (no renderer change — per the in-file Phase 11 comment). Remove the MOCKS/DEFAULT_STATE/`?state=` import and logic entirely.

    On DOMContentLoaded: resolve `#search-form`, `#search-input`, `#disclosure`, and `#results` (mount). Read settings once via `chrome.runtime.sendMessage({ type: 'getSettings' })` to get `sourceLanguage`/`targetLanguage`/`formality` (the worker getSettings case returns `{ settings }`). Attach a `submit` handler on `#search-form` that calls `event.preventDefault()` then `runSearch(input.value.trim())`. (Native form submit covers both Enter and the button — D-09.)

    Implement `async function runSearch(query: string)` performing the D-11 stage order:
    1. Guard: empty query → clear disclosure + results, return.
    2. Send `chrome.runtime.sendMessage({ type: 'searchTranslated', payload: { query, sourceLanguage, targetLanguage } })`. Await the SearchTranslatedResponse.
    3. As soon as the response arrives (translatedQuery known), build the disclosure input: `direct` true → `{ kind: 'direct', originalQuery: query }`; `translationFailed` true → `{ kind: 'failed', originalQuery: query }`; else `{ kind: 'translated', sourceLanguage, targetLanguage, originalQuery: query, translatedQuery }`. Set `disclosure.textContent = buildDisclosureText(input)` (textContent only — XSS-safe, read-only D-07). Then immediately `renderSerp({ kind: 'loading' }, document, mount)` to show the skeleton (D-11: disclosure before skeleton).
    4. If the response carried `{ error, kind }`: `renderSerp({ kind: 'error', errorKind: kind, message: error }, document, mount)` and stop (still showed disclosure if a translatedQuery was present; on a Brave error there may be no results).
    5. If `results` is empty array: `renderSerp({ kind: 'empty' }, document, mount)` and stop.
    6. Render RAW results: `renderSerp({ kind: 'populated', results }, document, mount)`.
    7. Overlay stage: build `buildBatchPayload(results)`, send `chrome.runtime.sendMessage({ type: 'translateBatch', payload: { items, config } })` where config = `{ sourceLanguage, targetLanguage, formality }`. On a successful TranslateBatchResponse with `translations`, `renderSerp({ kind: 'populated', results: mergeTranslations(results, translations) }, document, mount)`. On translateBatch error/timeout: leave the raw results in place (XLT-05 graceful degradation) — do NOT show an error over usable raw results.

    Re-submit (D-09): because the submit handler always calls runSearch fresh and each renderSerp call replaceChildren()-clears the mount, a new query re-runs the whole pipeline in place in the same tab — no page reload, no new tab. Guard against overlapping runs with a simple in-flight token (increment a module counter per run; ignore a stale overlay if a newer run started) so a slow overlay from a previous query never clobbers a newer search.

    Keep the file out of the node test harness (browser globals) — all unit-testable logic already lives in disclosure.ts / translate-batch.ts / serp-render.ts per the project test constraint.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user → search bar input | Untrusted query string entered on the page |
| page → worker | searchTranslated + translateBatch messages cross to the worker (keys never in payload) |
| worker reply → DOM | translatedQuery (disclosure) and Brave/LLM result strings cross back into the page DOM |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-06 | Tampering / XSS | disclosure line render | mitigate | translatedQuery + original query rendered into #disclosure via textContent only (never innerHTML); buildDisclosureText returns plain text. Acceptance grep enforces 0 innerHTML in disclosure.ts + search.ts. |
| T-11-07 | Tampering / XSS | SERP result render | mitigate | All Brave/LLM result strings rendered through renderSerp (Phase 9 textContent-only contract, SERP-03); search.ts never touches innerHTML. No new DOM-injection path introduced. |
| T-11-08 | Information disclosure | API keys on page | mitigate | Page only sends `{ query, sourceLanguage, targetLanguage }` (searchTranslated) and `{ items, config }` (translateBatch); never reads or sends Brave/LLM keys — keys stay in worker storage (XLT-01 / T-08-07). |
| T-11-09 | Denial of service | overlapping re-submits | mitigate | In-flight run token discards stale overlays so a slow previous-query overlay cannot clobber a newer search; renderSerp replaceChildren() bounds DOM growth. |
| T-11-SC | Tampering | npm/pip/cargo installs | mitigate | No new dependencies introduced by this plan. No install step. |
</threat_model>
