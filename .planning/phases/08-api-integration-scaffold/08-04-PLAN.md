<objective>
Add the Brave Search API key UI to the options page: a key input that persists to
`himeSettings.braveApiKey`, plus a separate "Test Brave Key" button with its own status line that
validates the key by routing a `testBraveKey` message through the background worker (D-04). To honor
XLT-01, the handler saves the key to storage first, then sends the message — the key is never carried
in the message payload, and the page-side test does not read/POST it directly the way the existing LLM
Test Connection does.

Purpose: SSET-01 (key field stored in chrome.storage) and SSET-02 (test/validate, "key required" when
absent). This is the only user-facing surface in Phase 8 — search page UI is Phase 11.

Output: Extended `src/options.html` and `src/options.ts`. No edits to test/unit.mjs (UI flow is
validated manually per 08-VALIDATION.md).
</objective>

| options page → background worker | testBraveKey message routes the validation through the worker; the key stays in storage |

| T-08-11 | Tampering | save-before-test race | mitigate | On save failure the handler surfaces the error and does not proceed to test against a stale key (Pitfall 5) |
