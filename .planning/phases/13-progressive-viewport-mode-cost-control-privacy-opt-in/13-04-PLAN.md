<objective>
Add the worker-side half of progressive mode to background.ts plus the panel
scroll-to-entry handling in sidepanel.ts: a `progressiveTranslate` handler that
resolves bytes, computes the AUTHORITATIVE content-hash dedup key over those bytes,
and funnels through the EXISTING `runImageJob`/`ocrAndTranslateImage` pipeline (one
funnel, two triggers — PROG-02); an `openImagePanel` handler that relays the
badge-click gesture into `chrome.sidePanel.open` plus a scroll-to-entry instruction
to the panel (D-04); and a `progressiveActivity` push so the content-side activity
count stays in sync. This plan builds NO new translation capability — it reuses
Phase 12's pipeline and its `storage.session` dedup map so a re-scrolled or
different-URL-same-bytes image is replayed from cache, never re-billed (PROG-03).

Purpose: Content-side dedup (Plan 03) is a cheap srcUrl first-filter; the
worker — the only place that resolves real bytes — must own the true content-hash
dedup so two URLs serving the same image never double-bill. And only the worker can
legally call sidePanel.open from a relayed gesture (PROG-06).

Output: three new message handlers + content-hash dedup in `src/background.ts`, and
scroll-to-entry handling in `src/sidepanel.ts`.
</objective>

| content script → worker | progressiveTranslate/openImagePanel payloads cross; only {srcUrl, dedupKey, tabId}-shaped data, never BYOK keys |
| worker → cloud (Vision + LLM) | image bytes + OCR text leave the device; keys read only here |
| relayed gesture → sidePanel.open | Chrome enforces gesture-eligibility; open must fire synchronously in the handler |
