<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM reply → content/worker parse | The model's batch reply is untrusted text; `parsePageBatchReply` is the parse boundary |
| Page DOM text → snapshot | Author-controlled page text feeds the walk; only `nodeValue` strings are read (no eval, no HTML interpretation) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-01 | Tampering | `parsePageBatchReply` (LLM reply → key map) | mitigate | Iterate `inputKeys` ONLY (never the parsed object's keys); accept only `typeof entry === 'string'`; reject array/non-object → `{}`. Cloned from `parseBatchReply` XLT-03. Test asserts an injected/extraneous key (`'evil'`/`'__proto__'`) is dropped. |
| T-15-02 | Elevation of Privilege | `page-walk.ts` module surface | mitigate | Module is pure: no `chrome.*`, no `document` writes, no `innerHTML`. Grep gate (`grep -c 'chrome\.'`, `grep -c innerHTML` → 0) enforces it. |
| T-15-03 | Information disclosure | `TranslatePageBatchMessage.payload` | mitigate | Payload carries source text + config only — never the BYOK key (key stays in worker storage, locked). Acceptance check: no `apiKey`/`key` field in the interface. |
</threat_model>
