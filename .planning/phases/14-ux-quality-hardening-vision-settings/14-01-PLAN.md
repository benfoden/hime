<objective>
Lay the worker-side data contract for Phase 14's UX hardening: a stable, dedup-keyed per-image number (`himeNum`, D-04 / backlog 999.4), a CJK/vertical detection flag on populated entries (D-03), and a graceful "image too large" classification path (D-03a) so oversized images surface the D-02 failure card instead of an opaque throw. This plan produces NO renderer/CSS changes — it only extends the types, the pure image-resolve math, the worker pipeline, and the test fixtures so the renderer plans (14-02, 14-03) and the page-language gate plan (14-02) have concrete fields to consume.

Purpose: Every visible Phase 14 affordance (the `[hime N]` chip, the CJK note, the named failure reason) needs a field carried from the worker through `deriveImageEntry` and the storage.session job map. This plan defines and populates those fields once, dedup-keyed, so a re-scroll/replay reuses the same number and the panel never invents data.
Output: New optional fields `himeNum` and `verticalOrCjk` on `ImageEntry`; `himeNum` on `ProgressiveBadgeMessage`; pure `isCjkLang` + `isOversizedForVision` helpers in `image-resolve.ts`; worker counter + flag + oversized wiring in `background.ts`; extended `panel-mock.ts` fixtures + new `test/image-resolve.mjs` cases.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Google Vision response → worker | Vendor-controlled OCR text + language code + error body crosses into the worker; consumed for the CJK flag (language code) and the named failure reason (error body). |
| worker → content/panel (messages) | himeNum + flags + reason text travel over `chrome.runtime`/`chrome.tabs.sendMessage`; rendered downstream textContent-only (other plans). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-01 | Information Disclosure | background.ts oversized/auth error message | mitigate | Reuse the existing `classifyError` path — the Google error body is surfaced as a classified message and the BYOK key lives only in the URL `?key=` searchParam, never in the thrown message (errors.ts:42-49 already guarantees this). New "image too large" message is worker-derived, no key, no user PII. |
| T-14-02 | Tampering | Vision-supplied `detectedLang` feeding isCjkLang | accept | The language code is used only for a boolean flag and a static caption (other plans); it is never eval'd, never used to build a request URL, and never reflected as HTML. Worst case is a wrong/absent caption — no security impact. |
| T-14-03 | Denial of Service | himeNum counter in storage.session | accept | Monotonic integer, session-scoped, bounded by per-page budget (10) and right-click frequency; no unbounded growth within a session, cleared on session end. |
| T-14-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs in this plan (pure TS edits + existing devDeps); no new dependency surface. |
</threat_model>
