# Phase 02.1: OpenRouter Provider Support - Context

**Gathered:** 2026-05-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Add OpenRouter as a third provider option in hime, enabling access to multiple LLM
backends (Claude, GPT, Llama, Mistral, etc.) through a single API key. OpenRouter uses
an OpenAI-compatible API format, so the implementation reuses the existing provider
abstraction with minimal new code. No new user-facing capabilities beyond provider selection.

</domain>

<decisions>
## Implementation Decisions

### Model selection (D-01)
- **D-01:** Fetch available models dynamically from OpenRouter's `/api/v1/models` endpoint.
  Populate the model dropdown when OpenRouter is selected as provider. Requires an API call
  on settings page load (when provider is openrouter) with error handling for network failures.
  No hardcoded model list — the dropdown is fully dynamic.

### API endpoint (D-02)
- **D-02:** Fixed endpoint: `https://openrouter.ai/api/v1`. No configurable base URL.
  OpenRouter uses OpenAI-compatible chat completions format (`/chat/completions`), so the
  provider implementation follows the same request structure as the OpenAI provider.

### Host permissions (D-03)
- **D-03:** Add `https://openrouter.ai/*` to `host_permissions` in manifest.json statically.
  Consistent with existing OpenAI and Gemini permission entries. No optional_permissions.

### Provider type union (D-04)
- **D-04:** Extend `ProviderConfig.provider` type from `'openai' | 'gemini'` to
  `'openai' | 'gemini' | 'openrouter'`. Update `PROVIDER_MODELS` to include an openrouter
  entry (empty array since models are fetched dynamically).

### Claude's Discretion
- OpenRouter-specific HTTP headers (e.g., `HTTP-Referer`, `X-Title` for rankings)
- Error classification mapping for OpenRouter-specific error codes
- Model list caching strategy (per-session vs per-page-load)
- Sort/filter of fetched model list in the dropdown

</decisions>
