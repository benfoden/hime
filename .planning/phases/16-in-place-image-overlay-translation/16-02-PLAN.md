<objective>
Extend the service worker so a single `translateImageBlocks` request resolves one image's bytes, OCRs it into per-paragraph boxes, batch-translates those blocks in ONE keyed-JSON round-trip, and replies with everything the content script needs to place overlays: `{ blocks:[{box,original,translated}], submitted:{w,h}, captureFallback }`. This requires two surgical changes to existing code: `GoogleVisionProvider.ocr()` must surface paragraph `blocks` (via the Plan 01 `collectParagraphBoxes`) alongside its unchanged `originalText`, and `downscaleAndGuard` must RETURN the submitted (post-downscale) dimensions it currently discards.

Purpose: The content script knows `naturalWidth` but NOT what the worker downscaled to — without the submitted dims, `mapBox`'s downscale-undo step is impossible (RESEARCH Pitfall 1). The worker is the only place that knows the submitted dims; this plan returns them. Per OVL-01. The v1.3 side-panel OCR path (D-04) stays intact.
Output: Extended `ocr()` + raw-response interfaces, widened `downscaleAndGuard`, a new `translateImageBlocks` worker case cloned from `translatePageBatch`, and an extended `test/vision-google.mjs`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| content script → worker (translateImageBlocks) | Untrusted page-origin message; carries only srcUrl/tabId/dedupKey/config — no key. |
| worker → Vision/LLM providers | Outbound BYOK calls; keys read from storage in the worker only. |
| Vision/LLM reply → worker parse | Provider JSON parsed defensively; LLM reply key-injection-guarded. |
| image bytes fetch (resolveImageBytes) | Only the user-page `<img>.src` is fetched (bounded SSRF surface). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-01 / T-16-02 | Information Disclosure | translateImageBlocks reply | mitigate | apiKey/googleKey read from `s.apiKeys`/settings in the worker ONLY; never placed in any sendResponse object; message payload has no key field (enforced by the Plan 01 type). |
| T-16-04 | Tampering | LLM keyed-JSON reply | mitigate | parsePageBatchReply iterates inputKeys only — model-invented keys (`__proto__`, etc.) never read (page-walk.ts:160-167). |
| T-12-12 / T-16-05 | Tampering (SSRF) | resolveImageBytes fetch | mitigate | Only the in-page `<img>.src` from the snapshot is fetched — not arbitrary worker input (background.ts:211-213); unchanged from v1.3. |
| T-16-06 | Information Disclosure | captureVisibleTab fallback geometry | mitigate | On capture-fallback the worker replies `{captureFallback:true}` with NO blocks, so unmappable screenshot geometry is never surfaced as a misleading overlay (Pitfall 2). |
| T-16-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs (milestone guardrail: zero new OSS deps). |
</threat_model>
