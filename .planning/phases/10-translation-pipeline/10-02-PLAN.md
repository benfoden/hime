<objective>
Wire the `translateBatch` message handler into the background service worker. The
case clones the existing `translate`/`searchTranslated` structure (resolve settings
→ guard provider+key → try/catch → `{ … }` | `{ error, kind }`), serializes the
page-supplied keyed `items` with `buildBatchTranslatePrompt`, calls
`provider.translate(...)` inside a `Promise.race` against an ~8s timeout (D-04 —
providers accept no external signal), validates the reply with `parseBatchReply`
(worker-side validation per RESEARCH Open Q1), and returns `{ translations }` on
success or `{ error, kind }` on failure.

Purpose: This is the only Chrome-coupled piece of Phase 10. It completes the
request/response side of XLT-02 and the worker half of XLT-05 (timeout/error never
blanks the page — the page keeps the raw stage). Live page-side sequencing (the two
sequential messages, debounce, query box) is Phase 11, out of scope here.

Output: `src/background.ts` (modified: +import, +`case 'translateBatch'`).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| page → worker (translateBatch payload) | `items` + `config` cross from the page into the worker; the API key never crosses (read from storage) |
| worker → LLM provider | Serialized keyed-JSON `items` are sent to the external LLM endpoint |
| LLM reply → worker → page | Untrusted model output returns and is validated before being handed back to the page |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01 | Information Disclosure | translateBatch prompt construction | mitigate | The worker serializes ONLY the page-supplied `items` (which carry just `t`/`d` — built by buildBatchPayload in 10-01). It never reads url/hostname from any SearchResult. XLT-04 holds end-to-end. |
| T-10-04 | Information Disclosure | API key in message | mitigate | The LLM API key is read from `chrome.storage` inside the worker (`s.apiKeys[s.provider]`); it is never part of the translateBatch message payload (existing XLT-01 architecture, mirrored from searchTranslated). |
| T-10-05 | Denial of Service | unbounded/slow LLM call | mitigate | Promise.race caps the call at ~8s (D-04). On timeout the worker returns `{ error, kind: 'network' }`; the page keeps its raw stage (XLT-05) — a slow model can never blank or hang the page. |
| T-10-03 | Tampering (prompt injection) | LLM reply | mitigate | parseBatchReply (validated in 10-01) runs worker-side: only well-formed `{t,d}` for original input keys are returned; injected/renamed keys are dropped, so a malicious reply cannot mismap or inject structural fields. |
| T-10-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this phase (RESEARCH Package Legitimacy Audit: not applicable). |
</threat_model>
