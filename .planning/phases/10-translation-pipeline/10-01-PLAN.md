<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| page → worker (translateBatch message) | Page-supplied `items` cross into the worker; in this plan only the type contract is defined (handler is Plan 02) |
| LLM reply → page render | Translated strings are untrusted model output that will be rendered on the SERP |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01 | Information Disclosure | `buildBatchPayload` | mitigate | Payload constructed from `{ t: title, d: description }` ONLY — url/hostname/faviconUrl are never referenced (XLT-04). Test 'XLT-04: serialized payload contains no result URL/hostname substring' asserts this structurally, not by convention. |
| T-10-02 | Tampering (XSS) | translated `title`/`description` in `mergeTranslations` output | mitigate | mergeTranslations treats translated strings as opaque data and places them in `title`/`description` of a `SearchResult`; the Phase 9 renderer assigns them via `textContent` (never innerHTML). Test asserts an `<img onerror>` payload survives as exact data here; the no-execution guarantee is the unchanged Phase 9 contract (SERP-03). |
| T-10-03 | Tampering (prompt injection) | LLM reply structure | mitigate | `parseBatchReply` validates structure (object, per-key `{t,d}` strings) and iterates the original `inputKeys`; an injected/renamed key cannot mismap onto a different result, and url/hostname are carried verbatim from the original SearchResult in mergeTranslations — injected content can only ever replace text that is rendered as plain text. |
| T-10-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this phase (RESEARCH Package Legitimacy Audit: not applicable). |
</threat_model>
